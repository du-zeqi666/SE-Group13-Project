## 四、项目背景讲解

### 4.1 单细胞测序数据是什么

PPT 中提到：

> 把组织中的单个细胞分离出来，分别测量每个细胞内部的分子信息，再通过测序读出这些信息。测序完成后，可以统计每个细胞中各个基因的表达量，从而得到一个“细胞 × 基因”的表达矩阵。

通俗来说，单细胞测序技术可以把一个组织中的很多细胞单独拿出来分析。每个细胞都会对应一组基因表达量，因此一个数据集可以表示为一个矩阵：

- **行：细胞**
- **列：基因**
- **矩阵中的值：某个细胞中某个基因的表达量**

例如，一个数据集有 100000 个细胞、20000 个基因，那么它就可以形成一个 `100000 × 20000` 的表达矩阵。每一行都可以看成一个高维向量，这个向量描述了一个细胞的分子特征。

### 4.2 为什么需要最近邻检索

在单细胞分析中，经常需要回答类似问题：

- 与当前查询细胞最相似的细胞有哪些？
- 这些相似细胞是否属于同一种细胞类型？
- 某个细胞在不同数据集或不同条件下有没有相似样本？
- 查询细胞附近的细胞群体分布如何？

这些问题本质上都可以转化为 **向量相似度检索问题**。  
即：给定一个查询向量，在大量细胞向量中找到距离最近或相似度最高的 Top-K 个向量。

### 4.3 为什么不能只用精确最近邻

PPT 中说明：

> 传统的精确最近邻搜索方法虽然能够返回准确结果，但在数据规模较大、维度较高的情况下，往往存在查询效率低、响应时间长、计算资源消耗大的问题。

精确最近邻检索通常需要把查询向量与所有细胞向量逐一比较。如果数据规模很大，例如几十万甚至上百万个细胞，每次查询都进行全量计算，系统响应速度会很慢。

因此，本项目引入 **近似最近邻检索（Approximate Nearest Neighbor, ANN）**。ANN 不一定保证每次都返回绝对精确的最近邻，但它可以在保持较高召回率的同时显著提高查询速度。

### 4.4 ANN 的意义

PPT 中提到：

> 近似最近邻检索（Approximate Nearest Neighbor, ANN）技术能够在保证较高检索质量的同时显著降低查询开销，因此已广泛应用于图像搜索、推荐系统、语义检索等场景，也适合用于单细胞数据。

因此，本项目的核心技术点是：  
**使用 ANN 技术在大规模单细胞高维向量中实现高效相似细胞搜索。**

### 5.3 索引构建模块

PPT 原文：

> 该模块负责根据输入数据建立检索索引，并完成索引的构建、保存和加载操作，为高效查询提供支持。

索引构建模块是 ANN 检索系统的核心之一。它的作用是把向量数据组织成适合快速查询的索引结构。

#### 基础功能

- 选择索引类型；
- 根据细胞向量构建索引；
- 保存索引文件；
- 加载已有索引；
- 显示索引状态。

#### 可支持的索引类型

PPT 中介绍了常见 ANN 算法：

1. **IVF（Inverted File Index）**
   - 先聚类，再在局部区域搜索；
   - 适合大规模向量检索；
   - 可以通过调整聚类数量和搜索范围控制速度与准确率。

2. **HNSW（Hierarchical Navigable Small World）**
   - 基于图结构构建近邻网络；
   - 高召回率；
   - 查询效率高；
   - 当前主流 ANN 方法之一。

3. **PQ（Product Quantization）**
   - 对向量进行压缩编码；
   - 可降低内存占用；
   - 适合超大规模数据。

#### 建议

- 使用 `FAISS` 实现 IVF、HNSW 或 Flat 精确检索；

### 6.3 数据处理与降维流程

PPT 中给出的常见流程为：

> 原始表达矩阵 → 质量控制（QC）→ 数据标准化 → 对数变换 → 高变基因筛选 → 数据缩放 → PCA 降维 → 生成低维细胞向量

逐步解释如下：

1. **原始表达矩阵**  
   原始矩阵记录每个细胞中每个基因的表达量。

2. **质量控制（QC）**  
   去除质量较差的细胞或异常基因，例如表达量过低、线粒体基因比例过高等情况。

3. **数据标准化**  
   消除不同细胞测序深度差异带来的影响。

4. **对数变换**  
   缓解表达量分布偏斜，使数据更适合后续分析。

5. **高变基因筛选**  
   选择变化较大的基因，减少无关特征，提高分析效率。

6. **数据缩放**  
   将特征调整到相近尺度。

7. **PCA 降维**  
   将高维基因表达向量降到较低维度，例如 20、50 或 100 维。

8. **生成低维细胞向量**  
   将降维后的结果作为 ANN 检索的输入向量。

PPT 中也提醒：

> 给大家提供的数据已经进行了处理，如果想要对数据进行其他的处理，可以在原始数据上再进行处理。这里使用的降维方法是 PCA，大家可以自行尝试自编码器、scVI 等方式，但是注意，盲目的使用其他方法生成 embedding 未必会让效果更好，请斟酌效果后使用。

因此，建议先使用课程提供的已处理数据完成系统主流程，再尝试其他 embedding 方法作为加分或扩展内容。

## 七、ANN 算法与检索库讲解

---

### 7.1 常见 ANN 算法

PPT 中列出了三类常见 ANN 算法。

#### IVF：Inverted File Index

PPT 原文：

> IVF（Inverted File Index）先聚类，再在局部区域搜索。

IVF 的基本思想是先把所有向量聚成若干簇。查询时先判断查询向量最接近哪些簇，只在这些局部区域中搜索，而不是扫描全部向量。

优点：

- 查询速度快；
- 适合大规模数据；
- 参数可调，便于做性能对比。

缺点：

- 需要训练聚类中心；
- 如果搜索簇数量太少，可能影响召回率。

#### HNSW：Hierarchical Navigable Small World

PPT 原文：

> HNSW（Hierarchical Navigable Small World）基于图结构构建近邻网络。高召回率，查询效率高，当前主流 ANN 方法之一。

HNSW 会把向量构建成近邻图，查询时在图中快速跳转，从而找到相似向量。

优点：

- 查询速度快；
- 召回率高；
- 实践中效果稳定；
- 适合课程项目快速实现。

缺点：

- 索引构建和内存占用可能较高；
- 参数需要调节。

#### PQ：Product Quantization

PPT 原文：

> PQ（Product Quantization）对向量进行压缩编码。特点：降低内存占用，适合超大规模数据。

PQ 的核心思想是压缩向量表示，减少内存占用。它适合数据量非常大的场景。

优点：

- 节省内存；
- 适合超大规模向量库。

缺点：

- 实现和理解难度相对更高；
- 可能带来一定精度损失。

---

### 7.2 ANN 检索库

PPT 中介绍了 FAISS 与 HNSWLIB。

#### FAISS

PPT 原文：

> Facebook AI Research 提供的高性能向量检索库。支持 GPU 加速，支持 IVF / PQ / HNSW，适合大规模向量搜索。

FAISS 是功能非常丰富的向量检索库，适合实现以下功能：

- 精确检索；
- IVF 检索；
- PQ 压缩检索；
- HNSW 检索；
- GPU 加速；
- 不同索引性能对比。

如果小组希望实现多种索引结构或做实验评估，FAISS 是比较合适的选择。

#### HNSWLIB

PPT 原文：

> 轻量级 ANN 检索库。查询速度快，内存效率高，部署简单。

HNSWLIB 更轻量，适合快速实现 HNSW 检索功能。它的优点是使用简单，学习成本较低。

#### 向量数据库

PPT 中提到：

> 向量数据库可以理解为：ANN 检索 + 数据库系统，除了检索外还支持元数据管理等数据库的功能。常见的还有 Milvus、Qdrant、Weaviate、ChromaDB 等。

向量数据库通常适合更完整的工程系统。对于课程大作业，如果小组想做得更像真实项目，可以考虑使用向量数据库；但如果时间有限，直接使用 FAISS 或 HNSWLIB 即可。

## 十七、中期展示讲解稿参考

中期展示可以按以下结构讲解。

### 17.1 开场

大家好，我们小组的大作业题目是“面向单细胞高维向量数据的近似最近邻检索系统”。本项目的目标是针对大规模单细胞数据，利用 ANN 技术实现高效的相似细胞检索，并通过 Web 页面完成查询和结果展示。

### 17.2 背景说明

单细胞测序数据可以表示为“细胞 × 基因”的表达矩阵，每个细胞都可以看成一个高维向量。当数据规模较大时，传统精确最近邻检索效率较低，因此我们使用近似最近邻算法提高查询速度。

### 17.3 已完成功能

目前我们已经完成了以下功能：

1. 单细胞数据读取；
2. 细胞向量提取；
3. ANN 索引构建；
4. Top-K 相似细胞检索；
5. 查询结果展示；
6. 初步 Web 页面演示。

### 17.4 系统演示

接下来展示系统界面。首先选择数据集，然后构建索引，输入一个查询细胞 ID，设置 Top-K 参数，点击查询后系统会返回最相似的若干细胞，并显示细胞信息和查询耗时。

## 二十、开发风险与注意事项

### 20.1 不要只做算法脚本

PPT 明确要求是 Web 应用，因此不能只提交一个 Python 检索脚本。必须有可以操作的页面或系统界面。

### 20.2 不要忽略 Git 提交记录

GitHub 提交记录是评分依据之一，应尽早创建仓库并持续提交。

### 20.3 不要把可视化留到最后

结项要求至少一种可视化结果。可视化往往需要和数据格式、查询结果联动，建议中期后尽早实现。

### 20.4 不要盲目更换降维方法

PPT 提醒自编码器、scVI 等方法不一定比 PCA 效果更好。基础功能完成前，不建议把大量时间放在复杂 embedding 方法上。

### 20.5 注意贡献度沟通

贡献度会影响个人得分，组长分配时需让组员知情并同意，避免分歧。

### 20.6 注意演示稳定性

展示和视频录制前应准备好固定数据、固定查询样例和备用方案，避免现场系统运行失败。

## 二十二、可写入报告的项目目标示例

本项目旨在设计并实现一个面向单细胞高维向量数据的近似最近邻检索系统。系统以单细胞表达矩阵或降维后的细胞向量为数据基础，通过 ANN 索引结构实现高效的相似细胞搜索。用户可以在 Web 页面中选择数据集、构建索引、输入查询细胞或查询向量，并设置 Top-K、距离度量和过滤条件等参数。系统返回相似细胞列表及其元信息，同时展示查询耗时、索引状态和可视化结果。通过该系统，可以提高大规模单细胞数据中相似样本检索的效率，并为后续细胞类型分析、跨数据集比较和单细胞数据管理提供支持。

## 二十三、总结

本次团队作业的关键不是单独实现某一个算法，而是完成一个具有工程完整性的单细胞 ANN 检索系统。项目应围绕以下主线展开：

> 单细胞数据读取 → 向量化表示 → ANN 索引构建 → Top-K 相似细胞检索 → 条件过滤 → 结果展示 → 性能评估 → Web 系统交付

中期阶段重点证明核心检索流程已经跑通；结项阶段重点展示系统完整性、可视化效果、性能分析、开发文档和演示视频。基础功能完成后，可以通过多数据集联合检索、ANN 算法改进或 RAG + 单细胞数据库等方向争取加分。
