# 环境问题与解决办法

本文档记录本项目在本地部署、启动和构建过程中已经实际遇到过的环境问题，以及对应的解决办法，便于后续成员排查。

---

## 1. 前端无法启动，提示找不到 react-app-rewired

### 现象

在 `frontend` 目录执行 `npm start` 时，出现类似报错：

```text
'react-app-rewired' 不是内部或外部命令，也不是可运行的程序或批处理文件。
```

### 原因

前端依赖没有安装完整，`node_modules` 缺失或安装过程中中断。

### 解决办法

先进入前端目录，再重新安装依赖：

```powershell
cd frontend
npm install
npm start
```

如果 `npm install` 因权限或缓存目录问题失败，可改用项目内缓存目录：

```powershell
cd frontend
npm install --cache .npm-cache --no-audit --no-fund
npm start
```

说明：安装成功后，可删除 `frontend/.npm-cache/`，避免将缓存目录误提交到仓库。

---

## 2. npm 安装依赖时报 EPERM / 无法写入缓存目录

### 现象

执行 `npm install` 时出现 `EPERM`、`operation not permitted`、`node_cache` 无法写入等报错。

### 原因

本机 npm 全局缓存目录位于受限路径，当前权限无法正常写入缓存文件。

### 解决办法

使用项目本地缓存目录完成安装：

```powershell
cd frontend
npm install --cache .npm-cache --no-audit --no-fund
```

安装结束后如果不再需要，可删除：

```powershell
Remove-Item .\frontend\.npm-cache -Recurse -Force
```

## 3. git pull --rebase 失败，提示存在未暂存修改

### 现象

执行 `git pull --rebase` 时，提示：

```text
error: cannot pull with rebase: You have unstaged changes.
error: Please commit or stash them.
```

### 原因

本地存在未提交或未暂存修改，常见情况包括：

- `frontend/package-lock.json` 因重新安装依赖而变化
- `frontend/.npm-cache/` 这类临时目录被生成

### 解决办法

先查看当前状态：

```powershell
git status --short
```

如果只是临时安装产物，可清理后再拉取：

```powershell
Remove-Item .\frontend\.npm-cache -Recurse -Force
git restore -- .\frontend\package-lock.json
git pull --rebase
```

如果这些改动是你自己需要保留的，则先提交或暂存：

```powershell
git add .
git commit -m "your message"
```

或：

```powershell
git stash
git pull --rebase
git stash pop
```

## 4. 前端构建索引失败，提示 Failed to build index. Check dataset validity.

### 现象

前端点击“构建索引”时提示：

```text
Failed to build index. Check dataset validity.
```

但数据集本身实际上是正常的，例如：

- 数组维度正常
- 数据类型为 `float32`
- 不含 `NaN` / `Inf`

### 原因

该问题不一定是数据无效。在 Windows 环境下，如果项目仓库路径中包含中文等非 ASCII 字符，FAISS 和 Annoy 在将索引文件写入磁盘时可能失败。

本项目已确认这种情况会导致：

- 数据加载正常
- 索引构建本身正常
- 但在“保存索引文件”时失败
- 前端最终只看到统一的笼统报错

### 当前解决办法

项目已将索引安全回退目录改为：

```text
D:/Users/Public/SE-Group13-Project/backend-storage/indices
```

也就是说：

- 如果仓库路径是纯英文路径，则优先写入仓库内的 `backend/storage/indices`
- 如果仓库路径包含非 ASCII 字符，则自动改写到上面的 D 盘安全目录

### 额外建议

如果后续仍遇到类似问题，可进一步检查：

1. 后端是否已重启，使最新配置生效
2. `D:/Users/Public/SE-Group13-Project/backend-storage/indices` 是否存在写入权限
3. 当前数据集是否确实已成功导入并生成 `.npy` 文件

## 5. 启动脚本显示已启动，但前端实际没有运行

### 现象

执行：

```powershell
.\scripts\start_project.ps1
```

脚本显示“启动命令已发送”，但浏览器无法访问前端页面。

### 原因

启动脚本只是发起进程，不一定能保证前端依赖已经安装完整，也不会自动拦截所有运行时失败。

### 解决办法

建议分开验证前后端：

后端：

```powershell
.\.venv\Scripts\python.exe .\backend\app.py
```

前端：

```powershell
cd frontend
npm start
```

如果前端启动失败，优先检查是否是依赖未安装或 npm 缓存权限问题。

## 6. PowerShell 无法激活虚拟环境

### 现象

执行虚拟环境激活脚本时，PowerShell 提示执行策略限制。

### 解决办法

可以先在当前终端临时放开执行策略：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy RemoteSigned
.\.venv\Scripts\Activate.ps1
```

如果仍不想依赖激活脚本，也可以直接使用虚拟环境中的 Python：

```powershell
.\.venv\Scripts\python.exe -m pip install -r .\backend\requirements.txt
.\.venv\Scripts\python.exe .\backend\app.py
```

## 建议排查顺序

当项目在本机运行异常时，建议按以下顺序检查：

1. 先看 `backend/.env` 是否填写正确，尤其是 MySQL 连接信息。
2. 再确认后端能否单独启动，健康接口 `http://127.0.0.1:5000/api/health` 是否正常。
3. 再确认前端依赖是否安装完整，`npm start` 是否能单独运行。
4. 如果是索引构建问题，再检查索引目录是否落在 `D:/Users/Public/SE-Group13-Project/backend-storage/indices`。
5. 如果是 Git 同步问题，再检查 `git status --short` 是否干净。
